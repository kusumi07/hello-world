# ソフトウェア工学 2024

本講義の内容をまとめていく。

## ソフトウェア工学の概説

ソフトウェアの定義とは
1. 実行されることによって必要な特性、機能、性能を提供する命令群
2. プログラムが適切に情報を扱うことを可能とするデータ構造
3. プログラムの操作や使用法を記述下情報
を満たすものである。

このソフトウェアを品質やコスト、納期の最適なバランスを実現するためにどのような開発すると良いのか考える学問がソフトウェア工学と呼ばれる。

ソフトウェアライフサイクルは最初にソフトに実現する機能・実現しない機能を明確にする要件定義をし、その要件定義を参照しながら設計書を作成する。その後、設計書に従ってシステム構築を行う製作を行い、製作されたものをテストしデバックを行う。システムが実際に運用されてからもソフトに問題がないか保守する。上記の流れに沿ってソフトウェアは作られる。

## プロジェクトマネジメント

プロジェクトとは有期性を持ち、独自性を持った仕事のことである。プロジェクトを達成するためにはフォアキャスティングやバックキャスティングの考えが使われることがある。フォアキャスティングはインプットを視点で物事を考えるが、目標が正確に到達できない可能性がある。バックキャスティングはアウトプット視点で物事を考え、目標が明確に設定されなと上手くいくことができないという特徴がある。

## 開発プロセス
### ウォーターフォール法
開発プロセスが段階的に進行するため前の工程が完了してから次のフェーズに進む。その為、変更に弱いデメリットがある。

### スパイラルモデル
開発プロセスが反復的に進行するため、要件を途中で見直すことが可能であるので柔軟に開発を進めることができる。プロジェクトの最初にプロトタイプを作成するが、その作成に想定外の作業量が発生するリスクがある。

### 反復型開発プロセス
開発システムが段階的に進行するが、ウォーターフォール法とは違い開発を複数の小さなサイクルに分けて進める。その為、要件の見直しや変更が可能な柔軟な対応を可能とし、各反復ごとに継続的な改善をすることができる。デメリットとしては作業量が増えることが挙げられる。

### アジャイルプロセス
開発システムを小さな反復であるスプリントに分け、各スプリントで機能を追加、改善していく。継続的なフィードバックを取り入れることができ、要件や計画が変更しても柔軟に対応することができる。代表的な例としてはスクラムやカンバンがある。

## UML

UMLとは万国共通の記法でオブジェクト指向プログラミングである。大きく分けると論理ビュー・プロセスビュー・開発ビュー・物理ビュー・ユースケースビューの五つあり、代表的なものは以下の通りである。

### ユースケース図
ユースケース図はシステムと外部のアクターとの関係を示す図であり、システムが提供する機能であるユースケースとそれを利用するアクターを視覚的に表現するものである。

### アクティビティ図
アクティビティ図は、システムやプロセスのワークフローや操作手順を表現するものである。

### クラス図
クラス図は、システムの性的構造を表現する。その為、クラスやその属性、メソッド、クラス間の関係を示すものである。

### オブジェクト図
オブジェクト図は、クラス図のインスタンス(具体的なオブジェクト)を示し、特定の時点におけるシステムの状態を表現するものである。

### シークエンス図
シークエンス図はオブジェクト間の動的なやり取りを時系列に表現をし、特定のシナリオにおけるメッセージの流れを示すものである。

### コンポーネント図
コンポーネント図は、システムの物理的な構成要素であるコンポーネントとそれらの依存関係を示すものである。

## コーディング

コーディングをする時にはコードを読む人が読みやすく、理解しやすいコードを書くことが重要である。その為のコーディングのルールが定まっている。例としては、pythonの場合は一行の長さを79文字、コメントの場合は72文字にすることやインデントするときはスペースキーを4回使ってから書き始めるなどルールが定まっている。pythonの場合はPEP8が主なルール書である。

このPEP8のルールを守ってコーディングができているのかを判断するチェックツールとしてflake8が存在しており、これを使うことで達成していない部分があればエラーとして表示してくれる役割を持つ。

## git
gitとは、分散型バージョン管理システムであり、かつてはオープンソースソフトウェア管理のためのソフトウェアであった。変更履歴が残るため、変更した箇所に戻ることも出来る。また、他人と共同編集することも出来る。同じファイルを触れていても、上書き保存されず、だれがどのような編集をしたかの履歴が全て残っている。

### 機能

#### バージョン管理
ソースコードのスナップショットを作成し、変更履歴として記録する

#### ブランチ管理
新しいブランチを作成し、異なる機能の開発を並行して行える

#### マージ
開発を終えたブランチをメインブランチに統合する

#### 差分の確認
変更された部分を確認し、どのファイルがどのように変更したかを比較する

#### プルとプッシュ
特定のリポジトリの変更を指定したリポジトリに反映させる

#### ログの確認
コミット履歴を表示し、誰が、いつ、どのような変更を行ったかを確認する

#### ステージングエリアの管理
変更されたすべてのファイルをステージングエリアに追加する

#### リバートとリセット
直前のコミットを取り消したり、それに伴う変更を完全に取り消す。

### コマンド

```
git add
```
特定のファイルの変更の変更の追跡を開始するために使用する ファイルをステージングエリアに追加する

```
git branch
```
新しいブランチを作成する

```
git checkout
```
指定されたブランチに切り替えて、作業ディレクトリを更新するまた、削除されたファイルを復旧する
#### オプション
"-b":付けることでディレクトリを作成しながら、作業ディレクトリを更新する

```
git clone
```
プロジェクトとすべてのバージョン履歴をダウンロードする

```
git commit
```
作業内容をスナップショットに保存する
#### オプション
"-m":メッセージを一度に指定する。"-a":自動的に変更されたファイルをステージングエリアに追加する。"--amend":履歴を変更することができる

```
git config --global user.name "<USER_NAME>"
```
Gitの設定においてユーザ名をフローバルに設定する

```
git config --global user.email "<USER_email>"
```
Gitの設定においてユーザのメールアドレスを設定する

```
git config --list
```
現在のGitの設定をリスト形式で表示する

```
git diff
```
まだステージされていないファイルの差分を表示する

```
git init
```
新しいGitリポジトリを作成する

```
git log
```
以前のコミットに関する情報を見ることができる

```
git merge
```
作業ディレクトリでの作業が終了したら使い、指定されたブランチの履歴を現在のブランチに統合する
#### オプション
"--abort":マージを施行する前の状態に復元する

```
git pull
```
ブックマークの履歴をダウンロードし、変更を統合する

```
git push
```
すべてのローカルブランチのコミットをアップロードする

```
git request-pull
```
変更の依頼をする

```
git reset
```
変更のステージングを解除

```
git revert
```
直前のコミットを元に戻す

```
git rm
```
ファイルを削除する

```
git stash
```
作業中の変更を一時的に保存し、作業ディレクトリをクリーンな状態に戻す

```
git status
```
コミット可能なすべての新規のファイルまたは変更のあるファイルを一覧で表示します

```
git symbolic-ref
```
指定されたブランチに現在のブランチを設定する

```
git --help
```
Gitでできることに関するヘルプを表示する
